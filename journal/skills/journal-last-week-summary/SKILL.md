# Journal Last Week Summary

## Overview

Create or fully regenerate the weekly summary for the previous completed Monday-Sunday week in this repo. Read the week's daily notes from `daily/`, follow linked or transcluded notes that contain the actual work details, summarize completed work, and suggest high-level team focuses for the following week. Write the result to `weekly/weekly-summary-<monday>.md`.

## Repo Conventions

- Daily notes are named `daily/YYYY-MM-DD.md`.
- Weekly summaries are named `weekly/weekly-summary-YYYY-MM-DD.md`, where the date is the Monday of the summarized week.
- Daily notes often contain `[[...]]` or `![[...]]` references to the real work notes for calendar blocks.
- Linked work notes frequently live in calendar-style folders such as `calendars/Focus Time/`, but search the repo rather than assuming a single location.
- Daily notes may also have a `# Where I'm Leaving Off` section with useful end-of-day context.
- Daily notes may contain `## Jira Ticket Candidates` sections generated by Codex. Treat those sections as suggested candidates only, not accepted tickets, committed sprint work, or authoritative backlog state without further supporting context.

## Workflow

1. Interpret "last week" as the previous fully completed Monday-Sunday week.
Use the Monday of that week for the target filename in `weekly/`.

2. Read all daily notes for that week from `daily/`.
If a day is missing, continue with the available notes and mention the missing coverage in the chat response.

3. Start with the daily note body.
Capture any substantive text directly in the daily note, especially the `# Where I'm Leaving Off` section.
If the daily note includes `## Jira Ticket Candidates`, use it only as a weak planning signal to help notice possible follow-ups. Do not summarize candidate tickets as accepted work unless other notes, Jira references, or explicit user/team context support that interpretation.

4. Extract wikilinks and transclusions from each daily note.
Resolve linked notes by exact filename search first. If there are duplicates, prefer the note whose date matches the daily note and whose path looks like a calendar or work-block note.

5. Read only the linked notes that add concrete work detail.
Prioritize notes that mention implementation, debugging, testing, reviews, investigations, support work, planning with concrete output, or hiring work that took meaningful time.

6. Ignore or compress low-signal material.
Skip lunch blocks, empty standups, generic calendar metadata, repeated fragments, and notes that only restate that work happened without saying what changed.

7. Build an internal workstream ledger before writing.
For each meaningful workstream, identify the current bet, the next proof or validation signal, unresolved risks or dependencies, and whether it needs team attention next week. Use this ledger to reason about suggested team focuses, but do not include a dense ledger table in the weekly note unless the user explicitly asks for it.

8. Synthesize the week into a concise summary plus team-focus suggestions and write it to `weekly/weekly-summary-<monday>.md`.
If the weekly file already exists, regenerate it from source notes and overwrite it rather than editing around the old text.

## Output Rules

- Use the heading format `# Weekly Summary: Month D-D, YYYY`.
- The summary body before `## Suggested Team Focuses` must contain at most 4 top-level bullet points total.
- Each top-level bullet must start with a very succinct title or one-line theme label.
- Each top-level bullet must contain exactly one sub-bullet that holds the supporting detail.
- Indent each sub-bullet with exactly 4 spaces.
- Each sub-bullet must be at most 2 sentences total.
- These are hard maximums, not targets. Use fewer bullets when the week is sparse.
- Group related work into themes. Prefer concrete outcomes and progress over a chronological list of meetings or time blocks.
- Keep the sub-bullet content concrete and information-dense; the top-level title should summarize, not replace, the detail.

After the summary bullets, add:

```md
## Suggested Team Focuses
```

- Include 3-5 bullets total.
- Each bullet should suggest a high-level focus for the entire team for the following week, not a personal Dylan task list.
- Base each suggestion on evidence from the week's notes plus the internal workstream ledger.
- Prefer workstream-level focuses such as milestone readiness, evaluation quality, review throughput, operational handoff, data quality, or dependency closure.
- Each bullet should include a concrete proof signal, for example a merged PR set, completed run sheet, documented operating procedure, dashboard, recorded execution attempt, or explicit unblocker decision.
- Do not invent commitments, owners, or deadlines that are not supported by the notes. Phrase uncertain items as suggestions.
- Keep each bullet to at most 2 sentences.

## Reading Strategy

Start from the daily notes. If a daily note mostly contains links, open the linked notes in small batches and stop following references once the notes become duplicative or low-signal.

Prefer extracting:

- finished work
- implementation progress
- bug fixes and debugging
- testing and validation
- design or planning with concrete decisions
- code reviews or operational support that materially affected the week

Avoid writing bullets like:

- Attended standup.
- Worked in the morning.
- Had a meeting with X.

Prefer bullets like:

- Finished robot client parallel-cell updates
    - Closed out the remaining reset and readiness checks needed for parallel cell operation.
- Investigated Codex sandboxing constraints
    - Identified the namespace setting required for Codex to run and noted the container-build follow-up.

For team focus suggestions, prefer bullets like:

- Generalist signs-of-life readiness
    - Keep team attention on the smallest end-to-end proof path: cell/network readiness, dummy observation/action exchange, transform confirmation, and one recorded execution attempt or clearly documented blocker.
- Evaluation repeatability and SQA handoff
    - Convert the week's ad hoc debugging into repeatable run sheets, operator guidance, and metrics artifacts so the next evaluation cycle can be SQA-driven rather than Dylan-driven.

## Practical Notes

- Prefer `rg --files` and exact filename search to locate linked notes.
- Use targeted reads with `sed -n` or similar instead of loading large files wholesale.
- Before writing, check the target weekly filename and overwrite it intentionally.
- Keep the weekly note concise even if the source notes are noisy.
